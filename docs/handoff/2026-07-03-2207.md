## 0. O que é este projeto (30 segundos)

**Generic Chronicle** é um mod de batalha para **Final Fantasy Tactics: The Ivalice Chronicles
(IVC)** — a remasterização de 2025 (Steam, `FFT_enhanced.exe`, engine "Faith" do FFXVI, x86-64,
protegida por **Denuvo**). Objetivo: a **Deep Combat Layer (DCL)** — fórmulas de combate
totalmente customizadas (dano, hit/miss/block/parry, status) onde o resultado depende de
atributos **E** equipamento **do atacante E do alvo**. NÃO é "deixar o jogo ultra-difícil"; é
substituir as fórmulas por um modelo autoral.

Entregue como um **code mod C# do Reloaded-II** (`ModId = fftivc.generic.chronicle.codemod`),
que instala hooks assembly no jogo e roda um poller gerenciado. Há também um **data mod**
separado (`fftivc.generic.chronicle`, sem ".codemod") que edita tabelas NXD/XML — mas fica
**DESLIGADO** durante toda investigação de code mod.

**A regra que domina o desenvolvimento:** Denuvo **virtualiza CÓDIGO, não DADOS.** Os cálculos
de combate rodam dentro de uma VM opaca, mas as structs/tabelas que a VM lê são memória normal
e escrevível. Todo controle vem de escrever os DADOS certos (input que a VM lê antes, ou output
staged que reescrevemos antes do apply). Onde a VM lê de uma tabela de dados, nós vencemos.

## 3. O estado atual — o MAPA DE LEVERS (tudo provado live, salvo indicado)

Este é o payoff. O controle de combate está **resolvido end-to-end**. Espinha de contexto por
(ação, alvo) = hook `computeActionResult 0x309A44` (rcx = order record em caster+0x1A0
[0]=slot,[1]=type,[2..3]=abilityId; dl = target idx; dispara em preview, charge, execução E IA).

| Camada | Lever | Como | Status |
|---|---|---|---|
| **Evasão de equipamento** (weapon/shield/accessory) | `ItemTableEvadeZero` | poke nas tabelas de item carregadas (VAs fixas escrevíveis) — a VM deriva daí | ✅ **PROVEN LT5-A4** |
| **Class evade** (job) | escrita live em `unit+0x4B` | cópia 1:1 no record; a VM lê live | ✅ proven 2026-06-27 |
| **Reações** (Shirahadori/Blade Grasp, Hamedo, Counter) | escrita de **Brave** `unit+0x2B` (10=suprime, floor de chicken; 100=força) OU strip R/S/M na data | o roll da reação é VM-interno; hook no roll REFUTADO | ✅ proven 2026-06-27 |
| **Dano/HP/MP** | pre-clamp `0x30A66F` (apply) + compute-point `0x281F8A` (`+0x1C4`) | reescrever staged antes do apply | ✅ proven (LT4) |
| **Status** | immunity `+0x5C` + master `+0x1EF`/`+0x61` | input bits a VM lê | ✅ proven (LT1/LT2) |
| **Hit% arbitrário** | o **MOD** computa a % e rola o **próprio RNG**, forçando o resultado binário pelos levers acima | a % da VM não é escrevível (globals recomputados) | design, desbloqueado |
| **Miss/parry/block VISUAL num hit** | paint kind `+0x1C0` + co-write Family-2 no finalize `0x2055FC` | LT5-C, refinamento de apresentação | desenhado, não testado live |

### Endereços das tabelas de item (o lever novo mais importante) — RVAs (some image base)

   Descobriram: o "poll racy" era um diagnóstico errado (não existe escritor por-ataque; os bytes
   são re-carimbados por copiadores de equip/refresh); a % da VM não é escrevível; o paint de

   das quais a VM deriva a evasão. Construí `ItemTableEvadeZero`. **LT5-A4 PASSOU: preview 100%,
   12/12 golpes conectando** contra alvos com 50% de parry/escudo. **Miss/block/parry RESOLVIDO.**
   Bônus: o "Writer A 0x59F550" era código de gamepad HID (falso positivo de byte-scan) — removido.

## 5. O modelo mental: a fronteira da VM

Tudo antes do "hit" (accuracy %, derivação de evasão de equipamento, roll de negação de reação)
é **VM-interno** e consome dados que a VM deriva. Os inputs de struct live que a VM honra:
**class evade `+0x4B`, Brave `+0x2B`, bytes de status/immunity.** Evasão de equipamento é honrada
só pela **tabela de itens** (dado que a VM lê). Regra geral: **onde a VM lê de um dado que
possuímos, vencemos; onde ela recomputa/rola internamente, controlamos pelo input que ela lê ou
pelo output staged que reescrevemos.** É o princípio `output-control-first` com a exceção de
input onde output não alcança.

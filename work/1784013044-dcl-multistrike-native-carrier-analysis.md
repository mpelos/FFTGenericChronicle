## DCL implementation boundary

- **Pummel — Strong:** formula `0x32` does not commit N native strikes. It computes
  `count = floor(random15 * X / 32768) + 1`, multiplies the one staged HP debit by that
  count, and sets one HP-debit result flag. The vanilla carrier is aggregate. DCL Pummel
  therefore needs an explicit managed strike loop to obtain per-strike contest and Guard
  depletion while retaining the authored once-per-action reaction policy.
- **Celestial Void / Corporeal Void / Dark Whisper — Strong:** these are the catalog records
  in this frontier with `RandomFire` set. Their damage formulas are mapped, but the protected
  outer consumer still owns repetition, target selection, and result-application order.
- **Nanoflare — Proven:** formula `0x5F` aliases the same single-result MA handler but the
  action has `RandomFire` clear. It is a single-hit MA action, not a multistrike mechanism.
- **Barrage — Strong:** formula `0x6A` delegates to the equipped-weapon formula and then the
  native normal-attack postprocessor. Weapon math is reusable; exact repeat count and commit
  cadence remain an integration/live gate.

## Remaining live gates

1. Observe the number and ordering of calculation-entry, selector, pre-clamp, and apply calls
   for one `RandomFire` action.
2. Observe Barrage repeat count, active-hand/weapon-formula selection, target stability, and
   whether each strike reaches the ordinary result apply separately.
3. Validate the managed Pummel carrier with one physical contest and Guard debit per authored
   strike, but no more than one reaction per Pummel action.

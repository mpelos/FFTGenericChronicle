# DCL active-weapon payload checkpoint

## Question

Determine whether the DCL can route a dual-wield strike through the exact weapon used by native
calculation without inferring the hand from result bytes or hit order.

## Static result

`computeActionResult` copies the 20-byte order record, loads `word[orderRecord+8]`, and retains it in
`SI`. The action-type-1 branch validates that value as protected lookup kind 6, resolves the weapon
row, reads its formula byte, and dispatches the formula. The value is therefore the exact selected
weapon item id.

Evidence is generated by `tools/analyze_dcl_active_weapon_payload.py` in
`work/1784078674-dcl-active-weapon-payload-analysis.md`. All ten executable anchors and five managed
integration checks pass against the installed Enhanced executable.

## Runtime integration

- The calc-entry ring stores `word[rcx+8]` in the existing 48-byte slot at `+36`.
- `DclActionContextCache` retains the payload with caster, target, action type, and ability id.
- `FormulaRuntimeContextBuilder` exposes `action.payload*`, exact `action.weapon.*`, and right/left
  match flags.
- `DclHitDecisionCache` includes the payload in exact lookup, consumption, invalidation, and finite
  defense commit keys. A mixed-weapon second strike cannot reuse the first weapon's decision.
- The weapon-skill mechanism profile selects the active left/right skill chain from the payload and
  derives damage type from exact `action.weapon.*` metadata rather than always using the right slot.

## Ambiguity rule

When right and left item ids differ, matching the payload to unit words `+0x20/+0x24` identifies the
side. When both item ids are equal, the side cannot be identified from the payload, but exact weapon
identity, category, formula, family, and skill grade are the same. DCL weapon routing therefore does
not need an artificial side guess in this case.

## Offline validation

- Code-mod smoke project builds with zero warnings and zero errors.
- Formula runtime smoke tests pass, including mixed Ninja Blade/Romandan Pistol selection and the
  identical-weapon ambiguity case.
- The weapon-skill mechanism profile validates with zero errors.
- The native/static analyzer reports every anchor and integration check as PASS.

## Remaining live boundary

Native Dual Wield cadence is not closed by this static proof. A controlled capture must still show
that both strikes reach separate result transactions and that each transaction carries the expected
payload. No additional offline hand-identity hypothesis is required before that test.

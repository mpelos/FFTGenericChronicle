## Native transaction read

```text
state 0x2F action cleanup
  -> queue 0x206344
     accepted (eax=1) -> state 0x29 and deliver the accepted native action
     empty    (eax=0) -> 0xD90CFA2
  -> post-chain cleanup 0x206050
  -> resume owner 0x205F28
  -> state 0x28
```

The same state-`0x2F` handler runs after the ordinary outer action and after each delivered
native Reaction. Therefore the first empty scan and the terminal empty scan after one or
more accepted Reactions converge at `0xD90CFA2`. The accepted pass-2 boundary cannot provide
this guarantee because it does not fire for an empty window and can fire several times in a chain.

At `0xD90CFA2`, `rbx` still owns the current execution actor and `eax == 0`; native actor/list
cleanup and the state-`0x28` write have not run yet. A managed hook may therefore acknowledge
only an already-payment-committed singleton canonical ActionInstance, then settle and retire it.
The actor can be the outer source on an empty first scan or the last reactor after a chain, so
source equality is not a valid completion guard; ownership must come from the pending canonical
ledger ticket and battle generation.

Battle-state global: `0xC6B1CC`. The boundary is static-proven and remains live-gated.
